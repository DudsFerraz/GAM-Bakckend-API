# Requirement: OpenAPI and Frontend API Documentation

## Status
Accepted

## Context
GAM needs a complete, discoverable, and versioned backend API contract that a Frontend Developer can use without reading backend implementation code or manually duplicating request and response types.

The backend and frontend remain in separate repositories. The backend therefore owns the authoritative OpenAPI contract, while the frontend consumes a version-pinned contract artifact and generates TypeScript types from it. This specification elaborates the OpenAPI responsibilities already established by `REQ-WEB-004`, `REQ-WEB-008`, `REQ-WEB-009`, `REQ-WEB-010`, and `REQ-WEB-011`.

## Ubiquitous Language

- `interactive request execution`: Swagger UI behavior that sends a real HTTP request to a running GAM backend, commonly exposed as **Try it out**.
- `live contract`: The OpenAPI document generated by a running backend and exposed through its documentation route.
- `release contract artifact`: The immutable `openapi.yaml` generated for and published with one backend release.
- `compatibility baseline`: The most recent explicitly accepted immutable release contract used as the `oasdiff` comparison target for subsequent changes.
- `baseline bootstrap`: The one-time approval path that establishes the first compatibility baseline when no prior release contract exists.
- `paged response`: A GAM-owned collection envelope containing items and page metadata.

## Functional requirements

### REQ-OPENAPI-001: Authoritative generated contract
The backend shall generate the authoritative OpenAPI 3.1 contract from implemented Spring controllers, request and response DTOs, validation constraints, security declarations, and explicit OpenAPI annotations by using Springdoc.

The live contract and release contract artifact shall be generated from the same backend definition. Developers shall not manually maintain a competing OpenAPI document.

Rationale:
A single code-first definition prevents the implemented HTTP API, interactive documentation, and cross-repository contract artifact from drifting into independent sources of truth.

Valid examples:
- A DTO validation constraint and its consumer-visible OpenAPI constraint change in the same backend change.
- CI exports `openapi.yaml` from the backend definition used by local Swagger UI.

Invalid examples:
- A developer edits a committed `openapi.yaml` independently of the controller contract.
- Swagger UI renders a document maintained separately from the release artifact.

---

### REQ-OPENAPI-002: Discoverable live documentation
The public API boundary shall expose unauthenticated read access to:

- `/api/docs` for Swagger UI; and
- `/api/openapi.json` for the live machine-readable contract.

Swagger UI shall allow interactive request execution in development environments. Production Swagger UI shall remain browsable but shall disable interactive request execution for every HTTP method.

The live contract shall include every GAM application endpoint reachable through the public `/api` boundary, including protected administrative and RBAC operations. Framework-generated routes, health and metrics routes, actuator routes, and private operational routes outside the frontend contract shall be excluded.

Rationale:
Frontend Developers need frictionless discovery in every enabled environment, while production documentation should not encourage accidental mutation of real data.

Valid examples:
- An unauthenticated developer can inspect a protected member endpoint and see that it requires bearer authentication.
- Development Swagger UI can call `GET`, `POST`, `PATCH`, and `DELETE` operations.
- Production Swagger UI displays operations without an **Execute** action.

Invalid examples:
- Reading the contract requires first obtaining a GAM Account.
- Production hides the entire contract as a substitute for endpoint authorization.
- Swagger documents only a selected subset of existing frontend-facing controllers.

---

### REQ-OPENAPI-003: Complete operation documentation
Every operation shall declare:

- one globally unique, explicit, lower-camel-case `operationId` formed from an action and resource;
- one primary consumer-facing capability or resource tag;
- a summary and purpose;
- its authentication and authorization requirements;
- path, query, header, cookie, and body inputs;
- complete success response schemas;
- every expected error status and error response schema;
- consumer-visible validation constraints; and
- realistic synthetic examples.

When relevant, an operation shall also document preconditions, side effects, retry or idempotency behavior, recommended use, discouraged use, and related operations.

Rationale:
Schema inference alone does not explain when or how a Frontend Developer should use an operation.

Valid examples:
- `approveMembershipSolicitation` describes its authorization, input, success response, expected conflicts, and side effects.
- A read-only operation omits retry prose when ordinary HTTP semantics already make it obvious.

Invalid examples:
- An operation relies on a generated Java method name as its stable identifier.
- A `409 Conflict` that is part of normal domain behavior is absent from the contract.
- Every operation contains meaningless boilerplate for concerns that do not apply.

---

### REQ-OPENAPI-004: Accurate consumer-visible schemas
OpenAPI schemas shall reflect every externally visible requirement governing a request or response field, including requiredness, omission, nullability, type, format, allowed values, length, range, pattern, and collection constraints.

An omitted property and a property supplied as `null` shall have distinct meanings unless the owning Requirement Specification explicitly makes them equivalent.

Examples shall use realistic synthetic GAM data and shall not contain real personal data, credentials, tokens, production hostnames, or copied production identifiers.

Rationale:
Generated TypeScript types and frontend validation are only useful when the contract accurately represents the API boundary.

Valid examples:
- A required non-null UUID is documented as a required string with `format: uuid`.
- An optional but non-null property may be omitted and rejects explicit `null`.

Invalid examples:
- A Bean Validation minimum is externally visible but missing from the schema.
- A real Member record is copied into an OpenAPI example.

---

### REQ-OPENAPI-005: Authentication contract representation
The OpenAPI contract shall define bearer access-token authentication as the default security requirement for protected endpoints and shall explicitly override that default for public endpoints.

The authentication operations shall document the CSRF header and browser-managed cookies required by the accepted browser-session requirements. The contract and API guide shall explain that a refresh token is browser-managed, is not entered into a Swagger authorization field, and is never exposed to JavaScript.

Rationale:
The contract must tell a Frontend Developer exactly which credentials and browser mechanisms each operation requires without weakening the accepted authentication model.

Valid examples:
- `GET /accounts/me` declares bearer authentication.
- Login declares no bearer requirement and documents `X-XSRF-TOKEN`.
- Refresh documents both CSRF proof and the browser-supplied `refreshToken` cookie.

Invalid examples:
- Every operation appears public because security is undocumented.
- Swagger instructs a developer to copy a refresh token into an authorization control.

### REQ-OPENAPI-006: Common API error envelope
Every JSON error response shall use this common envelope:

```json
{
  "timestamp": "2026-07-14T17:30:00Z",
  "status": 404,
  "code": "RESOURCE_NOT_FOUND",
  "message": "Member not found with the supplied identifier.",
  "details": {
    "resource": "Member",
    "identifier": "0190d5d4-52b3-7d30-a8d3-64b70d6c3142"
  }
}
```

`code` shall be the stable machine-readable discriminator for frontend behavior. `message` shall be human-readable and shall not be parsed for behavior. `details` shall contain structured error-specific context. `status` shall repeat the HTTP status for clients that retain the body independently of the response. `timestamp` shall support diagnostics.

The envelope shall not contain a redundant `error` reason-phrase field.

Rationale:
One structured envelope lets the frontend handle errors consistently while stable codes avoid brittle parsing of messages.

Valid examples:
- A validation error uses `code: VALIDATION_ERROR` and identifies invalid fields under `details`.
- A not-found error identifies the resource and identifier structurally.

Invalid examples:
- Frontend behavior depends on matching the English `message`.
- One controller returns an unrelated ad hoc error shape.
- The body repeats `Not Found` in an `error` field without adding information.

---

### REQ-OPENAPI-007: Pagination and sorting contract
Paged endpoints shall use zero-based page numbering. `page=0` shall identify the first page. The default page size shall be `20`, the maximum page size shall be `100`, and a request above the maximum shall fail with `400 Bad Request` rather than being silently clamped.

Sorting shall use repeatable `sort=field,direction` query parameters. Each endpoint shall document and enforce its allowed sort fields and default order. Unknown fields or directions shall fail with `400 Bad Request`.

Paged responses shall use a GAM-owned envelope rather than Spring's raw `Page` serialization:

```json
{
  "items": [],
  "page": 0,
  "size": 20,
  "totalElements": 0,
  "totalPages": 0,
  "first": true,
  "last": true
}
```

Filtering shall remain endpoint-specific and shall be documented independently of pagination and sorting. This specification shall not create a new universal filtering language.

Rationale:
A small stable envelope and explicit sort allowlists prevent frontend coupling to Spring serialization and persistence property names.

Valid examples:
- `?page=0&size=20&sort=surname,asc&sort=firstName,asc` uses two documented sort fields.
- An endpoint documents its structured search body separately from the query pagination controls.

Invalid examples:
- A response exposes Spring `pageable` internals.
- `size=5000` is silently reduced to `100`.
- Any JPA property name is accepted as a public sort field.

---

### REQ-OPENAPI-008: Common data representations
UUID identifiers shall use OpenAPI `type: string` and `format: uuid` and shall be treated as opaque values by clients.

Calendar dates shall use ISO 8601 `YYYY-MM-DD` values with OpenAPI `format: date`. Absolute timestamps shall use RFC 3339 UTC values ending in `Z` with OpenAPI `format: date-time`. The API shall not expose timezone-less date-time strings.

Enums shall be serialized as documented uppercase strings and never as ordinals. Removing or renaming an enum value shall be breaking. Adding a new response enum value shall be treated as potentially breaking for exhaustive frontend consumers.

Decimal fields shall document their serialization and constraints when such fields are introduced.

Rationale:
Explicit representations prevent timezone conversion errors, identifier coupling, and silent enum incompatibilities.

Valid examples:
- A birth date is `2026-07-14` and is not timezone-converted.
- An event instant is `2026-07-14T17:30:00Z` and may be displayed in the user's local timezone.

Invalid examples:
- A UUID's embedded bits are used to derive resource meaning.
- A response emits `2026-07-14T14:30:00` without an offset.
- An enum is serialized as `2`.

### REQ-OPENAPI-009: Local and release contract availability
Developers shall be able to browse the live contract during normal local development without publishing a backend release.

The repository shall provide one Maven Wrapper command that starts any required generation lifecycle, exports the generated contract, lints it, and reports contract changes without requiring developers to assemble those steps manually.

The generated `openapi.yaml` shall not be committed. Each immutable backend GitHub Release shall publish the matching `openapi.yaml`, and its OpenAPI `info.version` shall equal the backend release version.

Rationale:
Live development needs rapid feedback, while independently versioned repositories need an immutable release contract.

Valid examples:
- A developer consults `/api/docs` through the supported local frontend proxy.
- Backend release `1.2.0` publishes its matching `openapi.yaml` under tag `1.2.0`.

Invalid examples:
- A frontend build consumes a mutable contract from `main` or a `latest` URL.
- Developers must manually start, download, rename, validate, and compare the contract with unrelated commands.

---

### REQ-OPENAPI-010: Generated TypeScript types
The frontend shall use `openapi-typescript` to generate compile-time API types from an explicitly selected backend contract version.

Local frontend development may generate types from the live contract. Release and production builds shall generate or verify types against the pinned release contract artifact. Frontend code shall not maintain competing handwritten copies of backend DTO shapes.

The generated types shall not by themselves be treated as a runtime HTTP client or runtime response validator.

Rationale:
Generated types make contract drift visible during frontend compilation without coupling the backend decision to a particular runtime client library.

Valid examples:
- A renamed required response field causes the frontend type check to fail until usage is updated.
- The frontend chooses its HTTP client independently of type generation.

Invalid examples:
- A frontend interface manually duplicates every backend response DTO.
- A generated TypeScript declaration is assumed to validate an untrusted response at runtime.

---

### REQ-OPENAPI-011: Automated contract governance
Pull-request and release workflows shall generate the OpenAPI contract and lint it with a repository-owned Spectral ruleset.

The initial ruleset shall enforce the agreed consumer-facing essentials, including unique operation IDs, summaries and descriptions, tags, declared security, documented responses, examples, and valid schemas. It shall avoid unrelated generic style rules that do not improve GAM integration.

Every API change shall produce an `oasdiff` report against the target contract. Breaking changes shall fail the contract check by default.

The target contract shall be the configured compatibility baseline. Normal pull-request, branch, and release checks shall fail closed when no compatibility baseline is configured.

When the project has no prior release contract, a maintainer may explicitly invoke the baseline bootstrap from the default branch. The bootstrap shall still generate and lint the contract, shall publish it under a new immutable release tag, and may omit `oasdiff` only because no earlier contract exists. The bootstrap shall be unavailable after a compatibility baseline is configured.

After each accepted backend release publishes its release contract artifact, maintainers shall promote that artifact's immutable URL as the compatibility baseline for subsequent changes. Promotion shall occur only after the release has passed the existing compatibility checks. A pull-request artifact, mutable branch URL, failed release, or current candidate contract shall not become the baseline.

During pre-production, an explicitly approved breaking change may proceed without a legacy compatibility layer when the pull request records Developer approval and confirms coordinated frontend contract and type updates. After production begins, breaking changes shall follow `REQ-WEB-010`.

Rationale:
Generated documentation remains trustworthy only when completeness and compatibility are checked automatically.

Advancing the baseline preserves changes accepted after the first release. If release 1 adds an operation and release 2 removes it, comparing release 2 only with an older contract that predates the operation would not detect that removal.

Valid examples:
- A pull request shows that a response property became required and blocks until explicitly handled.
- A description-only improvement is visible without being misclassified as a runtime break.
- The first approved contract is bootstrapped once, and the resulting immutable release URL is configured as the baseline.
- After release `1.2.0` passes against the previous baseline, its `openapi.yaml` becomes the baseline for work targeting the next release.

Invalid examples:
- Contract freshness depends only on a developer remembering to open Swagger UI.
- A breaking response change is published without a visible contract diff.
- Every candidate is compared forever with the initial contract even after later releases add new operations and schemas.
- A workflow compares a candidate contract with itself or promotes a pull-request artifact before release acceptance.

---

### REQ-OPENAPI-012: Endpoint documentation definition of done
Creating or modifying an endpoint, request or response shape, validation constraint, security requirement, status code, error response, pagination rule, or externally visible side effect shall include the corresponding OpenAPI documentation change in the same work.

The endpoint change shall not be complete until the generated contract is reviewed, Spectral passes, and the `oasdiff` result is handled according to `REQ-OPENAPI-011`.

The repository shall maintain:

- an OpenAPI documentation guideline for documentation authors;
- an OpenAPI software guideline for endpoint implementers; and
- a step-by-step OpenAPI developer workflow for generation, browsing, linting, diffing, and TypeScript generation.

Rationale:
OpenAPI must evolve as part of endpoint work rather than as deferred documentation cleanup.

Valid examples:
- A new validation annotation is accompanied by an accurate schema constraint and example.
- A response status change updates annotations, generated contract, and the reviewed diff together.

Invalid examples:
- A controller change is merged with a promise to document it later.
- A generated schema exists but lacks the business error responses required by the endpoint.

## Acceptance scenarios

```gherkin
Scenario: Frontend Developer discovers a protected endpoint
  Given the GAM API documentation is enabled
  When an unauthenticated Frontend Developer opens /api/docs
  Then the protected endpoint is visible
  And its bearer security requirement is documented
  And no GAM Account is required merely to browse the contract

Scenario: Production documentation cannot execute requests
  Given Swagger UI is running in production
  When a developer inspects any operation
  Then the operation documentation is visible
  And interactive request execution is unavailable

Scenario: Generate frontend types during local development
  Given a backend exposes the live OpenAPI contract
  When the frontend runs openapi-typescript against that contract
  Then TypeScript request and response types are generated
  And no backend release is required

Scenario: Reject an undocumented breaking change
  Given a pull request changes an existing API response incompatibly
  When CI generates and compares the OpenAPI contract
  Then oasdiff reports the breaking change
  And the contract check fails until the change is explicitly handled

Scenario: Publish a version-matched release contract
  Given backend release 1.2.0 passes contract checks
  When the release is published
  Then its GitHub Release contains openapi.yaml
  And the contract info.version is 1.2.0

Scenario: Detect removal of a contract added after the initial baseline
  Given an accepted release added an API operation
  And that release contract was promoted as the compatibility baseline
  When a later candidate removes the operation
  Then oasdiff reports the breaking removal
  And the contract check fails by default
```

## Open questions

* What plugin execution and test-profile configuration will implement `.\mvnw.cmd -Popenapi verify` while keeping that developer command stable?
* What exact release workflow will authenticate the frontend repository when fetching a contract from a private GitHub Release, if the repositories are private?

## Out of scope

* Redoc, Scalar, or another additional documentation renderer.
* Schemathesis or another schema-driven property-testing framework.
* AsyncAPI and message-driven interfaces.
* A handwritten API changelog.
* A universal filtering language.
* Selecting a frontend runtime HTTP client.
* Runtime TypeScript response validation.
* Money, file-transfer, or rate-limiting policies before GAM exposes those concerns.
* Static shared credentials or credentials that work in production.
* URL-versioning the API with `/v1`.
